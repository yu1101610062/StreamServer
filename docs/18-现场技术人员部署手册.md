# StreamServer 现场部署手册

本文面向现场技术人员，用于在 Linux AMD64 服务器上部署或升级 StreamServer 离线安装包。现场不需要源码仓库，也不需要联网拉取业务镜像。

## 1. 适用范围

适用安装包：

```text
streamserver-offline-v0.1.0-linux-amd64-cpu-only-<日期>.tar.gz
streamserver-offline-v0.1.0-linux-amd64-gpu-enabled-<日期>.tar.gz
```

目标服务器要求：

- 操作系统为 Linux，CPU 架构为 `x86_64/amd64`
- 磁盘空间充足，至少能同时存放离线包、解压目录、Docker 镜像和运行数据
- 需要能使用 `root` 用户，或具备等效 `sudo` 权限
- 如果使用 GPU 角色，服务器必须已经安装 NVIDIA 驱动，并且 `nvidia-smi` 可正常执行

Docker 要求：

- 如果服务器已有可用 Docker 和 Compose，安装脚本会自动跳过 Docker 安装
- 如果服务器没有 Docker，且离线包内包含 `tools/docker/offline-docker-*.tar.gz`，安装脚本会自动安装 Docker/Compose/Buildx
- 如果服务器已有 Docker 但处于损坏、权限不可用或版本混乱状态，需先由现场运维处理；安装脚本不会强制修复已有环境

建议现场使用 `root` 执行安装，避免 Docker 安装后当前用户没有 Docker socket 权限。

## 2. 部署前信息收集

部署前先确认以下信息。

| 项目 | 示例 | 说明 |
| --- | --- | --- |
| 安装角色 | `all-in-one-host-cpu` | 单机部署通常选 all-in-one，分布式部署按中心和工作节点拆分 |
| 安装目录 | `/home/streamserver` | 安装器默认目录，不会自动追加角色子目录；同一台服务器多个部署不能共用目录 |
| Compose 项目名 | `ss-aio` | 同一台服务器多个部署不能重复 |
| 服务器业务 IP | `192.168.1.20` | 用于页面访问、媒体访问和节点上报 |
| 主网卡 | `eth0` | 普通 HTTP/gRPC/API 流量使用 |
| 组播网卡 | `eth1` | 组播输入/输出使用；只有一张网卡时可与主网卡相同 |
| PostgreSQL 端口 | `5432` | control-plane/all-in-one 会占用 |
| Core HTTP 端口 | `8080` | 页面和 HTTP API |
| Core gRPC 端口 | `50051` | agent 连接 core |
| Agent HTTP 端口 | `8081` | agent 健康检查和内部接口 |
| ZLM HTTP/RTMP/RTSP 端口 | `80/1935/554` | 媒体播放和推流 |
| Hook/API 密钥 | 自动生成或现场指定 | 多节点部署时中心和工作节点必须一致 |
| 鉴权模式 | 关闭或本地账号密码 | 本地账号密码模式需设置管理员账号 |

部署前检查端口占用：

```bash
ss -lntp
```

如果默认端口已被占用，安装时按提示改用其他端口。

## 3. 安装角色选择

CPU-only 包可选：

- `control-plane`：只安装控制面和数据库，适合中心节点
- `worker-host-cpu`：只安装 CPU 工作节点和 ZLMediaKit，适合独立媒体节点
- `all-in-one-host-cpu`：单机完整部署，包含控制面、数据库、CPU 工作节点和 ZLMediaKit

GPU-enabled 包额外可选：

- `worker-host-gpu`：GPU 工作节点和 ZLMediaKit
- `all-in-one-host-gpu`：单机完整部署，并让工作节点使用 GPU

常见选择：

- 单机演示、单机联调、单机组播验证：选 `all-in-one-host-cpu`
- 多服务器部署：中心服务器选 `control-plane`，媒体服务器选 `worker-host-cpu` 或 `worker-host-gpu`
- 有 GPU 转码需求：选 GPU 包中的 GPU 角色

## 4. 安装步骤

将离线包和校验文件上传到目标服务器，例如：

```bash
/opt/install/streamserver-offline-v0.1.0-linux-amd64-cpu-only-20260602.tar.gz
/opt/install/streamserver-offline-v0.1.0-linux-amd64-cpu-only-20260602.tar.gz.sha256
```

进入目录：

```bash
cd /opt/install
```

校验安装包：

```bash
sha256sum streamserver-offline-v0.1.0-linux-amd64-cpu-only-20260602.tar.gz
cat streamserver-offline-v0.1.0-linux-amd64-cpu-only-20260602.tar.gz.sha256
```

确认两边 SHA256 一致。若 `.sha256` 文件中带有原打包机路径，直接对比 hash 值即可。

解压：

```bash
tar -xzf streamserver-offline-v0.1.0-linux-amd64-cpu-only-20260602.tar.gz
cd streamserver-offline-v0.1.0-linux-amd64-cpu-only-20260602
```

执行安装：

```bash
./install.sh
```

安装脚本会自动执行以下流程：

1. 检查服务器是否为 Linux AMD64
2. 检查 Docker 和 Compose 是否可用
3. 如 Docker 不可用且包内有 Docker 离线包，自动安装 Docker/Compose/Buildx
4. 询问安装角色
5. 询问安装目录、项目名、端口、网卡、IP、密钥和鉴权配置
6. 导入离线镜像
7. 写入二进制、前端文件、配置文件、Compose 模板和本地目录
8. 可选安装 systemd 开机自启动服务
9. 可选立即启动服务

如果提示是否立即启动，通常选择 `Y`。

## 5. 安装后检查

进入安装目录。示例：

```bash
cd /home/streamserver
```

查看服务状态：

```bash
./bin/streamserver-compose ps
```

期望看到对应服务为 `Up` 或 `healthy`。单机 CPU 部署通常包含：

```text
postgres
media-core
zlmediakit
media-agent
```

检查控制面健康状态：

```bash
curl -fsS http://127.0.0.1:8080/health/ready
```

期望返回：

```json
{"status":"ready", ...}
```

检查工作节点健康状态：

```bash
curl -fsS http://127.0.0.1:8081/health/ready
```

检查 ZLMediaKit：

```bash
curl -fsS "http://127.0.0.1:80/index/api/getStatistic?secret=<HOOK_SHARED_SECRET>"
```

`HOOK_SHARED_SECRET` 可在安装目录 `.env` 中查看。

浏览器访问：

```text
http://<服务器IP>:8080/
```

如果安装时启用了本地账号密码鉴权，使用安装过程中设置的管理员账号登录。

## 6. 常用运维命令

进入安装目录后执行：

```bash
./bin/streamserver-compose ps
./bin/streamserver-compose logs -f
./bin/streamserver-compose logs --tail=200 media-core
./bin/streamserver-compose logs --tail=200 media-agent
./bin/streamserver-compose restart media-core
./bin/streamserver-compose restart media-agent
./bin/streamserver-compose down
./bin/streamserver-compose up -d
```

如果安装时启用了 systemd 服务：

```bash
systemctl status ss-<项目名>.service
journalctl -u ss-<项目名>.service -f
systemctl restart ss-<项目名>.service
```

实际 unit 名以安装脚本输出为准。

## 7. 目录说明

安装目录典型结构：

```text
/home/streamserver/
├── compose.yml
├── .env
├── bin/
│   ├── media-core
│   ├── media-agent
│   ├── streamserver-config
│   └── streamserver-compose
├── ui/
├── certs/
├── data/
│   ├── postgres/
│   ├── media/work/
│   ├── media/logs/
│   └── zlm/www/
├── zlm/
└── docs/
```

重要目录：

- `bin/`：服务二进制，后续小版本升级可只替换对应二进制
- `ui/`：控制台前端静态文件
- `data/postgres/`：数据库数据，禁止随意删除
- `data/media/work/`：媒体处理工作目录
- `data/zlm/www/`：ZLM HTTP 根目录和媒体产物目录
- `.env`：运行配置，升级前应备份

## 8. 升级方式

### 8.1 原地升级

适用于完整离线包升级。

1. 上传新离线包到服务器
2. 解压新离线包
3. 执行新包的 `install.sh`
4. 选择与原部署相同的安装角色
5. 安装目录填写原安装目录
6. Compose 项目名保持原值
7. 按提示确认原地升级

安装脚本会备份原目录中的：

```text
.env
compose.yml
docker-compose.yml
bin/
ui/
zlm/
docs/
```

备份目录格式：

```text
<安装目录>.backup-<日期时间>
```

原地升级不会主动删除 PostgreSQL 数据目录，也不会清空业务历史数据。

### 8.2 只替换 media-core

适用于仅更新控制面逻辑，不更新镜像、agent、ZLM、PostgreSQL 和 UI。

```bash
cd /home/streamserver
cp bin/media-core bin/media-core.bak-$(date +%Y%m%d%H%M%S)
cp /path/to/new/media-core-linux-amd64 bin/media-core
chmod 755 bin/media-core
./bin/streamserver-compose restart media-core
```

`media-core` 启动时会自动执行数据库迁移。旧历史数据不会因为替换二进制自动删除。

## 9. 常见问题处理

### Docker 不可用

现象：

```text
Docker 不可用
缺少 Compose 命令
```

处理：

1. 确认本离线包中是否包含 `tools/docker/offline-docker-*.tar.gz`
2. 推荐使用 `root` 重新执行 `./install.sh`
3. 如果服务器已有 Docker 但不可用，先检查：

```bash
docker --version
docker info
docker compose version
docker-compose version
systemctl status docker
```

如果已有 Docker 正在运行但版本或权限异常，不建议安装脚本强行覆盖，需先由现场运维确认处理方案。

### 服务一直 unhealthy

查看状态：

```bash
./bin/streamserver-compose ps
```

查看日志：

```bash
./bin/streamserver-compose logs --tail=200 media-core
./bin/streamserver-compose logs --tail=200 media-agent
./bin/streamserver-compose logs --tail=200 zlmediakit
./bin/streamserver-compose logs --tail=200 postgres
```

常见原因：

- 端口被占用
- `.env` 中 IP、端口或密钥配置错误
- PostgreSQL 密码或数据目录权限异常
- GPU 角色缺少 NVIDIA 驱动或 Docker nvidia runtime
- `media-core` 首次升级时正在创建数据库索引，需等待一段时间后再复查

### 页面打不开

检查：

```bash
curl -v http://127.0.0.1:8080/health/ready
ss -lntp | grep ':8080'
```

如果本机可访问、外部不可访问，检查防火墙、安全组或现场网络策略。

### 工作节点未上线

检查：

```bash
./bin/streamserver-compose logs --tail=200 media-agent
grep -E 'AGENT_CORE_ENDPOINT|AGENT_NODE_ID|HOOK_SHARED_SECRET' .env
```

多节点部署时确认：

- 工作节点能访问中心节点 `CORE_GRPC_PORT`
- 工作节点和中心节点 `HOOK_SHARED_SECRET` 一致
- 每个工作节点 `NODE_ID` 不重复
- 主网卡和组播网卡选择正确

### ZLM Hook 异常

检查 `.env`：

```bash
grep -E 'HOOK_SHARED_SECRET|ZLM_API_HOST|ZLM_HTTP_PORT' .env
```

检查 ZLM 日志：

```bash
./bin/streamserver-compose logs --tail=200 zlmediakit
```

检查 core 日志：

```bash
./bin/streamserver-compose logs --tail=200 media-core
```

## 10. 现场交付确认

交付前至少确认以下项：

- `./bin/streamserver-compose ps` 中服务状态正常
- `media-core` ready 接口返回 `ready`
- 如有工作节点，`media-agent` ready 接口返回 `ready`
- 控制台页面可打开
- 本地账号密码模式下管理员账号可登录
- ZLM 统计接口可访问
- 安装目录、`.env`、Hook 密钥、管理员账号交接给现场负责人
- 如配置了 systemd，重启服务器后服务能自动恢复

建议保存以下输出作为交付记录：

```bash
uname -a
docker --version
docker compose version || docker-compose version
cd /home/streamserver
./bin/streamserver-compose ps
curl -fsS http://127.0.0.1:8080/health/ready
```
