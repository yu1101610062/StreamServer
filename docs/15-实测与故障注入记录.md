## 3. 本轮实测前的代码与脚本修正

本轮实测过程中发现并修正了 3 个真实问题。

### 3.1 `multicast_bridge` 的 live 输入直出 MPEG-TS 组播会因为 DTS/PTS 异常失败

现象：

- `multicast_bridge` 在 `RTSP -> UDP MPEGTS multicast` 且 `process.mode = passthrough` 时，FFmpeg 直接 `-c copy` 会报：
- `Timestamps are unset in a packet`
- `Non-monotonous DTS`
- `first pts and dts value must be set`
- `av_interleaved_write_frame(): Invalid data found when processing input`

结论：

- 问题不在 Docker 组播网络。
- 问题也不在探测脚本。
- 根因是从 ZLM 拉出的 live RTSP 包时间戳不适合直接 remux 到 MPEG-TS。

修正：

- 对精确场景 `live 输入 + UDP MPEGTS 组播输出 + passthrough` 增加稳定化 fallback。
- 策略为：视频重编码、音频保持 `copy`，用视频重编码重新生成稳定时间戳。

落点：

- `crates/media-agent/src/runtime.rs`

### 3.2 E2E 源样本时长不足，后半段故障注入可能跑到源任务自然结束

现象：

- `file_to_live` 源任务在前半段链路没问题，但在长时间故障注入序列里可能先自然结束。

修正：

- 把 E2E 样本时长从 180 秒调整为 600 秒。

落点：

- `scripts/e2e/compose-e2e.sh`

### 3.3 `rtp_receive` 实测失败是测试脚本读错了事件 JSON 路径

现象：

- `rtp_server_opened` 事件已经入库。
- 但脚本等待 `local_port` 时始终读不到，导致误报失败。

根因：

- Core 持久化 Agent 事件时包了一层 `payload`。
- 脚本直接查询 `local_port`，没有兼容 `payload.local_port`。

修正：

- 测试脚本查询任务事件字段时，先查顶层路径，再查 `payload.*` 路径。

落点：

- `scripts/e2e/compose-e2e.sh`
