# 17. 离线部署打包与安装

## 1. 目标

本文件定义一套从 `macOS arm64` 或 `Linux` 机器打包、再到 `Linux AMD64` 目标机离线解压部署的流程。

离线包目标：

- 打包机为 `macOS arm64` 或 `Linux` 主机
- 目标机为 `Linux AMD64`
- 目标机不需要联网拉取 StreamServer、PostgreSQL、ZLMediaKit 相关镜像
- 一个总压缩包内同时包含：
  - `media-core`
  - `media-agent`
  - PostgreSQL
  - ZLMediaKit
  - 角色化 Compose 模板
  - 单一入口安装脚本
  - 部署说明文档

## 2. 前提假设

本方案默认以下前提已经满足：

- 打包机已安装并启动 Docker（macOS 通常为 Docker Desktop，Linux 通常为 Docker Engine）。
- 完整打包需要具备 `docker buildx` 能力来构建 `linux/amd64` 镜像；仅生成骨架包时可配合 `--skip-images` 跳过。
- 目标机已安装 Docker Engine，且 `docker compose` 或 `docker-compose` 任一可用。
- 目标机为 `Linux AMD64`，且具备足够的磁盘空间存放镜像归档和运行数据。

说明：

- “完全离线”指应用镜像和部署模板都随包提供，不再从公网拉镜像。
- Docker Engine 本身不在本包内分发，因为它依赖目标 Linux 发行版。

## 3. 打包命令

在仓库根目录执行，脚本会交互式询问是否生成 `GPU-enabled` 离线包：

```bash
./scripts/package-offline-bundle.sh
```

可选参数：

```bash
./scripts/package-offline-bundle.sh --output-dir /path/to/dist
./scripts/package-offline-bundle.sh --with-gpu
./scripts/package-offline-bundle.sh --without-gpu
```

如果打包机在国内网络环境，脚本默认会使用：

- Debian APT 镜像：`http://mirrors.tuna.tsinghua.edu.cn`
- GPU 镜像 Ubuntu APT 源：默认保留官方源
- Cargo sparse 镜像：`https://rsproxy.cn/index/`

也可以在执行前显式覆盖：

```bash
APT_MIRROR=http://mirrors.aliyun.com \
UBUNTU_APT_MIRROR=http://mirrors.aliyun.com \
CARGO_REGISTRY_MIRROR=sparse+https://rsproxy.cn/index/ \
./scripts/package-offline-bundle.sh
```

如果希望回退到官方源，可将对应环境变量置空：

```bash
APT_MIRROR= \
UBUNTU_APT_MIRROR= \
CARGO_REGISTRY_MIRROR= \
./scripts/package-offline-bundle.sh
```

脚本行为：

1. 交互式选择生成 `CPU-only` 还是 `GPU-enabled` 离线包；也可以通过 `--with-gpu` / `--without-gpu` 显式指定。
2. 在 `macOS arm64` 或 `Linux` 主机上通过 `docker buildx` 构建 `linux/amd64` 的 `media-core`、`media-agent` 运行镜像；若选择 `GPU-enabled`，则额外构建 `media-agent-gpu` 运行镜像。
3. 优先复用本地已有的 `linux/amd64` PostgreSQL 和 ZLMediaKit 镜像；本地不存在或平台不匹配时再联网拉取。
4. 将所需镜像导出为离线 `tar`。
   - `CPU-only` 包不包含 `media-agent-gpu-linux-amd64.tar`
   - `GPU-enabled` 包包含 `media-agent-gpu-linux-amd64.tar`
4. 生成一套离线测试用自签名证书。
5. 复制安装脚本、Compose 模板、ZLM 模板和本文档。
6. 输出最终 `tar.gz` 压缩包与对应 `sha256` 校验文件；包名会明确包含 `cpu-only` 或 `gpu-enabled`，并使用构建日期（如同日重复打包则自动追加序号）。

解压后目录结构类似：

```text
│       ├── media-core.pem

    └── common/
```

## 5. 目标机安装方式

将 `tar.gz` 拷贝到目标 Linux AMD64 主机后：

```bash
tar -xzf streamserver-offline-v0.1.0-linux-amd64-<cpu-only|gpu-enabled>-<yyyymmdd>[-<n>].tar.gz
cd streamserver-offline-v0.1.0-linux-amd64-<cpu-only|gpu-enabled>-<yyyymmdd>[-<n>]
./install.sh
```

安装脚本会交互式询问安装角色：

- `control-plane`
  - 安装 `media-core + PostgreSQL`
  - 适用于中心控制节点
  - `media-core` 使用 `host` 网络；PostgreSQL 保持 bridge
  - 只暴露控制面 HTTP/gRPC 端口，不承载媒体组播
- `worker-host-cpu`
  - 安装 `CPU-only media-agent + ZLMediaKit`
  - 适用于所有工作节点，尤其是组播和需要直接绑定宿主机网卡的场景
  - 安装时会分别询问主网卡和组播网卡；普通流量默认走主网卡，组播输入/输出默认绑定组播网卡
  - `media-agent` 会共享挂载 `./data/zlm/www`，用于写入可 HTTP 访问的文件产物
  - 会直接占用宿主机媒体端口，更适合专用媒体节点
- `worker-host-gpu`
  - 安装支持 NVIDIA GPU 的 `media-agent + ZLMediaKit`
  - 适用于转码、重编码推流等希望优先走 CUDA/NVENC 的工作节点
  - 只让 `media-agent` 获取 GPU；`ZLMediaKit` 仍按普通 host 模式运行
  - 安装前会检查 `nvidia-smi` 和 Docker `nvidia` runtime
- `all-in-one-host-cpu`
  - 单机安装 `media-core + PostgreSQL + CPU-only media-agent + ZLMediaKit`
  - 适用于同一台机器同时验证控制面和真实组播
  - 安装时会分别询问主网卡和组播网卡；如果只有一张网卡，两者可以选择同一张
  - `media-core`、`media-agent` 和 `ZLMediaKit` 使用 `host` 网络；PostgreSQL 仍留在 bridge 网络
  - `media-agent` 会共享挂载 `./data/zlm/www`，用于写入可 HTTP 访问的文件产物
  - 会直接占用宿主机 `8080/50051/8081/80/554/1935` 等端口
- `all-in-one-host-gpu`
  - 单机安装 `media-core + PostgreSQL + media-agent + ZLMediaKit`，并让工作节点部分支持 NVIDIA GPU
  - 适用于单机联调、单机 GPU 转码验证和真实组播验证
  - 只让 `media-agent` 获取 GPU；`media-core` 仍使用普通单一版本

说明：

- `CPU-only` 离线包不会包含 `worker-host-gpu`、`all-in-one-host-gpu` 模板，也不会在安装器中提供 GPU 角色选项。
- `GPU-enabled` 离线包会同时包含 CPU-only 与 GPU-enabled 两组模板。

模式选择建议：

- 只有当媒体服务必须直连宿主机网卡时，才选择 `all-in-one-host-cpu` 或 `all-in-one-host-gpu`

安装脚本会完成以下动作：

1. 校验目标机是否为 `Linux AMD64`
2. 校验 `docker` 与 Compose 命令可用
3. 交互式采集角色所需参数
   - `worker-host-cpu` / `worker-host-gpu` / `all-in-one-host-cpu` / `all-in-one-host-gpu` 会列出可用 IPv4 网卡，支持输入编号或网卡名，并要求分别选择主网卡和组播网卡
   - 上述 4 类带工作节点的角色还会询问“额外节点标签”；安装器默认只写入一个算力标签：CPU 机型默认 `cpu`，GPU 机型默认 `gpu`，用户输入的标签会去重后追加到 `AGENT_LABELS`
   - `control-plane` / `all-in-one-host-cpu` / `all-in-one-host-gpu` 会询问是否启用 `media-core` 内建用户名密码鉴权
   - 若启用，会继续采集管理员用户名和密码，并自动生成 JWT 签名密钥对
4. `docker load` 导入该角色需要的离线镜像
5. 复制一套自签名测试证书到安装目录
6. 若启用本地鉴权，会在安装目录的 `certs/auth/` 下生成一套 Ed25519 JWT 密钥
7. 生成 `compose.yml`、`.env` 和 `zlm/config.ini`
8. 创建本地数据目录
9. 若启用本地鉴权，会临时启动 PostgreSQL 并自动执行 `media-core auth bootstrap-admin`
10. 提示是否先用正式证书替换自签名证书
11. 可选立即执行 `docker compose up -d` 或 `docker-compose up -d`

说明：

- 离线安装器当前支持两种开箱模式：
  - 关闭鉴权
  - 启用 `local_password` 本地用户名密码鉴权
- 工作节点默认不再自动写入 `worker`、`all-in-one`、`host`、`offline` 这些角色/部署形态标签；如果调度策略需要这些标签，请在安装时手动追加
- 若要使用 `external_jwt`，仍需在安装完成后手工调整 `.env` 和外围接入配置。

## 6. 安装后目录

按角色安装后，目标目录通常类似：

```text
/opt/streamserver/<role>/
├── compose.yml
├── .env
├── certs/
│   ├── self-signed/
│   ├── auth/
│   └── custom/
├── data/
│   ├── postgres/
│   ├── media/work/
│   ├── media/logs/
│   └── zlm/www/
├── zlm/
│   └── config.ini
└── docs/
    └── 17-离线部署打包与安装.md
```

常用管理命令：

```bash
cd /opt/streamserver/<role>
docker compose -f compose.yml up -d
docker compose -f compose.yml ps
docker compose -f compose.yml logs -f
docker compose -f compose.yml down
```

若现场机器只有 Compose v1，请将以上命令中的 `docker compose` 替换为 `docker-compose`。

## 7. 证书、HTTPS 与 mTLS

离线包内会预置一套自签名测试证书：

- `certs/self-signed/ca.pem`
- `certs/self-signed/media-core.pem`
- `certs/self-signed/media-core.key`
- `certs/self-signed/media-agent.pem`
- `certs/self-signed/media-agent.key`
- `certs/self-signed/https.pem`
- `certs/self-signed/https.key`

用途：

- `media-core.pem` / `media-core.key`
  - 用于 `media-core` 的 gRPC mTLS 服务端证书
- `media-agent.pem` / `media-agent.key`
  - 用于 `media-agent` 连接 `media-core` 时的客户端证书
- `https.pem` / `https.key`
  - 给前置反向代理测试 HTTPS 时使用
- `certs/auth/jwt-ed25519-private.pem` / `jwt-ed25519-public.pem`
  - 仅在安装时启用 `local_password` 鉴权时生成
  - 用于 `media-core` 本地登录模式下签发和验证 access token

默认策略：

- HTTPS 默认关闭
- gRPC mTLS 默认关闭
- 安装脚本会提示：如果你已有正式证书，应该优先替换自签名证书

替换建议：

- 将正式证书放到安装目录的 `certs/custom/`
- 修改 `.env` 中对应的证书路径
- 如果需要先处理证书，安装脚本会停止自动启动，等待你完成替换

### 7.1 mTLS 开启方法

`media-core` 侧启用：

```env
CORE_GRPC_TLS_CERT_PATH=/certs/custom/media-core.pem
CORE_GRPC_TLS_KEY_PATH=/certs/custom/media-core.key
CORE_GRPC_TLS_CLIENT_CA_PATH=/certs/custom/ca.pem
```

`media-agent` 侧启用：

```env
AGENT_CORE_ENDPOINT=https://<control-plane-host>:50051
AGENT_CERT_PATH=/certs/custom/media-agent.pem
AGENT_KEY_PATH=/certs/custom/media-agent.key
AGENT_CA_PATH=/certs/custom/ca.pem
AGENT_TLS_DOMAIN_NAME=streamserver-core.local
```

说明：

- 若只做离线测试，可把 `/certs/custom/` 替换成 `/certs/self-signed/`
- `AGENT_TLS_DOMAIN_NAME` 用来匹配内置测试证书中的服务端域名 `streamserver-core.local`

### 7.2 HTTPS 开启方法

当前项目自身没有内置 HTTPS HTTP listener，`media-core` 仍监听 HTTP `8080`。

如需 HTTPS，请在前置反向代理中终止 TLS，再转发到 `media-core:8080`。可选方式包括：

- Nginx
- Caddy
- Traefik

离线包中的 `https.pem` / `https.key` 只用于开发或内网测试。正式环境应替换成正式证书。

## 8. 端口约束

当前离线包模板固定了 ZLMediaKit 的默认对外端口语义：

- HTTP: `80`
- RTMP: `1935`
- RTSP: `554`

原因：

- 当前 `media-core` 播放地址生成逻辑默认按标准端口生成 RTSP/RTMP 地址。
- 如果把 ZLM 暴露到非默认 RTSP/RTMP 端口，控制面当前生成的播放 URL 会不准确。

因此离线包模板默认保持：

- `media-core`: `8080 / 50051`
- `media-agent`: `8081`
- `ZLMediaKit`: `80 / 1935 / 554`

补充说明：

- `all-in-one-host-cpu`、`all-in-one-host-gpu`、`worker-host-cpu` 和 `worker-host-gpu` 会直接绑定宿主机媒体端口，部署前应先确认端口未被占用
- `file_transcode` 的输出路径现在必须位于 `/data/zlm/www/artifacts/transcode/...`，这样产物才能复用工作节点的 ZLM HTTP 服务直接访问
- GPU 模式只支持 NVIDIA，且仅 `media-agent` 使用 GPU；`control-plane` 不区分 GPU/CPU 版本

## 9. 多工作节点部署建议

推荐部署方式：

1. 在中心节点执行一次 `control-plane` 安装。
2. 在每个工作节点分别执行一次 `worker-host-cpu` 或 `worker-host-gpu` 安装。
3. 每个工作节点使用不同的 `NODE_ID` 和 `AGENT_NODE_NAME`。
4. 所有工作节点的 `HOOK_SHARED_SECRET` 必须与中心控制节点保持一致。

建议：

- 组播场景优先 `worker-host-cpu`
- GPU 转码或重编码推流场景优先 `worker-host-gpu`
- 普通联调和标准代理场景也统一使用 `worker-host-cpu`
- 单机组播验证场景优先 `all-in-one-host-cpu`
- 单机 GPU 联调或 GPU 转码验证场景优先 `all-in-one-host-gpu`
- 生产组播节点建议为组播流量准备独立物理网卡；若现场只有一张网卡，主网卡和组播网卡可以选同一张

## 10. ZLM Hook 与 API 安全

离线模板里默认采用同一个密钥同时用于：

- ZLM `[api].secret`
- StreamServer `HOOK_SHARED_SECRET`

这样做的目的只是降低离线部署复杂度。

若进入正式生产，建议后续再拆分：

- ZLM HTTP API secret
- Hook shared secret

另外：

- `control-plane` 模板默认允许空 `HOOK_SOURCE_ALLOWLIST`
- `all-in-one-host-cpu` / `all-in-one-host-gpu` 模板默认保留空 allowlist，避免 host 网络下因源地址变化误伤正常 Hook 通信
- `worker-host-cpu` / `worker-host-gpu` 场景建议先依赖 shared secret，再按现场网络情况补充 allowlist

## 11. 升级方式

升级建议步骤：

1. 在新的源码版本上重新执行打包脚本，得到新的离线包。
2. 将新离线包复制到目标机。
3. 进入现有安装目录，先执行：

```bash
docker compose -f compose.yml down
```

4. 用新包重新运行 `install.sh`，指向同一个安装目录。
5. 确认 `.env` 中已有关键配置未被误改。
6. 启动：

```bash
docker compose -f compose.yml up -d
```

如果现场机器只有 Compose v1，则将这里的 `docker compose` 替换为 `docker-compose`。

`media-core` 启动时会自动执行数据库迁移，因此控制面升级时不需要单独再跑一次迁移脚本。

## 12. 设计边界

当前离线包方案刻意不覆盖以下内容：

- Docker Engine 本体安装
- 生产级反向代理和 HTTPS 终止
- `macvlan` 网络自动编排
- ZLM 非默认 RTSP/RTMP 端口下的播放 URL 修正

这些能力仍建议按现场运维方案单独处理。
