下面给你一版**最终定稿**，按“**只用 ZLMediaKit 开源版**、**Rust 做统一控制面**、**必须支持 UDP 组播**、**可以直接开工**”来写。🙂

## 1. 最终结论

最终方案定为：**`Rust media-core + Rust media-agent + ZLMediaKit OSS + FFmpeg/ffprobe + PostgreSQL`**。
边界也明确下来：

* **ZLMediaKit 开源版**负责实时媒体面：接入、代理、分发、录制、RTSP 组播、RTP/GB28181 接收与发送、REST API、Web Hook。
* **FFmpeg**负责复杂媒体处理：离线转码、文件转直播、固定 UDP 组播输入/输出、复杂滤镜、多输出。
* **Rust**负责统一控制面：任务、模板、默认值解析、并发调度、恢复、权限、审计、Web 管理台。
  这样做的原因是，ZLMediaKit 开源版本身已经提供了多协议 MediaServer、RESTful API、Web Hook、RTSP `rtp multicast`、RTP/GB28181、录制等能力；但官方 README 同时把“任意音视频转码”和“RTSP-TS/HLS/HTTP-TS/RTP multicast/UDP multicast ingestion”等更宽的能力列在**闭源专业版**的 Transcoding/IPTV 版本下面，所以开源版方案不能把这些能力当作基础前提。([ZLMediaKit][1])

FFmpeg 官方文档也明确把它定义成通用媒体工具：可以有任意数量输入/输出，`tee` muxer 可以“一次编码，多路输出”，而实际支持哪些协议、格式、编码器/解码器，取决于 build，可通过 `-protocols`、`-formats`、`-encoders`、`-decoders` 查看。([FFmpeg][2])

## 2. 总体架构

```text
上层业务系统
   │  REST/JSON
   ▼
media-core (Rust)
  ├─ API Gateway
  ├─ Task Service
  ├─ Template & Default Resolver
  ├─ Scheduler / Lease Manager
  ├─ Recovery Engine
  ├─ ZLM Hook Receiver
  ├─ Stream Registry
  ├─ Record Catalog
  ├─ Web Console Backend
  └─ Auth / Audit / RBAC
   │
   ├──────────────► PostgreSQL
   │
   ├──────────────► ZLM Adapter ─────► ZLMediaKit
   │                                  ├─ 拉流代理 / 转发
   │                                  ├─ 协议分发
   │                                  ├─ 录制
   │                                  ├─ RTSP 组播
   │                                  ├─ RTP/GB 接入与发送
   │                                  └─ REST API / Web Hook
   │
   └──────────────► Agent RPC ───────► media-agent (Rust)
                                      ├─ FFmpeg Supervisor
                                      ├─ ffprobe Preflight
                                      ├─ File Manager
                                      ├─ Multicast NIC Manager

```

这个拆分直接对应现有能力面：ZLM 已公开 `addStreamProxy`、`addFFmpegSource`、`startRecord/stopRecord`、`openRtpServer`、`listRtpServer`、`startSendRtp`、`addStreamPusherProxy`、`getMediaList`、`getThreadsLoad`、`getWorkThreadsLoad`、`getStatistic`、`kick_session`、`getMediaPlayerList` 等接口，也提供 `on_publish`、`on_record_mp4`、`on_stream_none_reader`、`on_stream_not_found`、`on_server_started`、`on_server_keepalive`、`on_rtp_server_timeout` 等 Hook，所以它非常适合做“被 Rust 编排的数据面”。([ZLMediaKit][3])

## 3. 组件详细设计

### 3.1 `media-core`

`media-core` 是唯一对上暴露的服务，职责是：

* 北向 API：任务创建、查询、控制、节点管理、记录查询。
* 参数编译：把用户传入的 `TaskSpec` 解析成 `ResolvedTaskSpec`。
* 调度：挑选节点、发租约、发起执行、重试、回收。
* 恢复：服务重启、节点重启、ZLM 重启后的自动对账。
* 事件中心：接收 ZLM Hook、agent 心跳、FFmpeg 生命周期事件。
* Web 管理台后端：任务页、流页、录像页、组播页、调试页。
* 审计和鉴权：RBAC、操作日志、幂等。

技术栈建议用 **Tokio + Axum + SQLx**。Tokio 官方明确说明它更适合 I/O bound 场景，CPU 密集任务不该直接塞进 runtime；Axum 设计上就是跑在 Tokio/Hyper 上，并且直接整合 tower 中间件生态；SQLx 是异步、纯 Rust、带连接池，支持 PostgreSQL 和 SQLite。([Tokio][4])

### 3.2 `media-agent`

每个媒体节点部署一个 `media-agent`，职责是：

* 启停 FFmpeg/ffprobe 进程；
* 采集本地 CPU / 内存 / 磁盘 / GPU / NIC 资源；
* 维护本机组播网卡绑定策略；
* 上报能力矩阵；
* 接管恢复中的本地进程；
* 推送 FFmpeg stdout/stderr、退出码、进度、产物信息。

我建议 `media-core` 与 `media-agent` 用 **gRPC** 通讯，原因不是“更潮”，而是它更适合日志流、进度流、节点心跳和强类型契约。这个是架构选择，不依赖特定第三方产品。

### 3.3 节点能力探测

每个 `media-agent` 启动后要自动做两件事：

1. 探测 **FFmpeg build 能力**：执行 `ffmpeg -protocols`、`-formats`、`-encoders`、`-decoders`，把结果写入 `node_capabilities`；
2. 探测 **ZLM 能力**：调用 `/index/api/version`、`/index/api/getApiList`，记录版本和可用接口。

这样调度器在分配任务时，不是“拍脑袋假设这个节点支持某协议/编码器”，而是根据真实 build 能力做匹配。FFmpeg 官方文档对这些枚举选项有明确说明。([FFmpeg][2])

### 3.4 ZLMediaKit 节点

ZLM 节点只做媒体面，不做业务逻辑。
它负责：

* 实时拉流代理、推流代理；
* 多协议分发；
* RTSP 会话组播；
* MP4/HLS 录制；
* RTP/GB28181 接收 / PS-RTP 发送；
* 会话查看、踢会话、抓图、统计。

开源版能力覆盖 RTSP/RTMP/HLS/HTTP-FLV/HTTP-TS/fMP4/MP4/GB28181/RTP；RTSP 明确支持 `rtp over udp / tcp / http / multicast`；对常见编码，官方列出的稳定集是 H264/H265/AAC/G711/OPUS，部分路径支持 MJPEG，而“其他编码可以转发但不能做协议转换”。([ZLMediaKit][1])

## 4. 统一任务模型

上层永远只操作“任务”，不直接操作 ZLM API，也不允许提交任意 FFmpeg 命令。
统一模型：

```json
{
  "type": "stream_ingest | stream_bridge | file_transcode",
  "name": "string",
  "priority": 50,
  "common": {},
  "input": {},
  "stream": {},
  "expose": {},
  "process": {},
  "publish": {},
  "record": {},
  "recovery": {},
  "schedule": {},
  "resource": {}
}
```

系统内部会生成两份数据：

* `requested_spec`：上层原始请求；
* `resolved_spec`：补齐默认值之后的最终参数。

**调度、恢复、重试、审计全部只看 `resolved_spec`**。这点非常关键，否则默认值一变，老任务重放就会出现行为漂移。

### 4.1 任务类型

V1 北向只定义 3 类：

* `stream_ingest`：把实时源或离线源接入为平台内部流，可录制、可暴露播放协议；
* `stream_bridge`：把源桥接到显式目标，只负责文件或组播输出；
* `file_transcode`：离线/VOD 转码任务，只产出文件。

内部执行器仍可区分 ZLM proxy、RTP server、FFmpeg 进程，但这不再体现在北向 `task_type` 上。

## 5. 参数模型与默认值

参数分 10 组：`common / input / stream / expose / process / publish / record / recovery / schedule / resource`。
上层可以只传最少字段，系统会靠模板和默认值自动补齐。

### 5.1 `common`

任务名、租户、幂等键、优先级、操作人、最大重试次数、启动超时、停止宽限期、回调地址、标签。

### 5.2 `input`

支持的 `kind`：

* `rtsp`
* `rtmp`
* `hls`
* `http_flv`
* `http_ts`
* `http_mp4`
* `file`
* `udp_mpegts_multicast`
* `rtp_multicast`
* `gb_rtp`

新增：

* `source_mode = live | vod`

其中 `file`、`http_mp4` 固定为 `vod`；`rtsp`、`rtmp`、`http_flv`、组播、`gb_rtp` 固定为 `live`；`hls`、`http_ts` 必须显式指定。
其中 `input.kind=file` 时，`input.url` 表示相对 `/data/media/work` 的路径；前导 `/` 会被自动去掉，但不允许使用 `..` 路径穿越或完整 URL。

组播类附加字段：

* `group`
* `port`
* `interface_ip`
* `ttl`
* `reuse`
* `pkt_size`
* `dscp`
* `buffer_size`
* `fifo_size`
* `probe_timeout_ms`

这些都对应 FFmpeg UDP 协议栈已有能力，如 `localaddr`、`reuse`、`ttl`、`pkt_size`、`dscp` 等。([FFmpeg][5])

### 5.3 `process`

* `mode = passthrough | copy_or_transcode | force_transcode`
* `bitrate / fps / gop`
* 缩放、裁剪、旋转、去隔行、降噪
* 水印、字幕、音轨映射、静音补齐
* 截图、抽帧、封面、拼接、切片

说明：

* 北向接口不再手动指定 `video_codec / audio_codec / profile / preset`
* 具体是走 CPU 还是 GPU、使用 H.264 还是 HEVC，由控制面和工作节点根据任务语义、输入编码和节点能力自动决定

### 5.4 `stream`

仅 `stream_ingest` 使用：

* `app`
* `name`
* `vhost`

### 5.5 `expose`

仅 `stream_ingest` 使用：

* `enable_rtsp`
* `enable_rtmp`
* `enable_http_ts`
* `enable_http_fmp4`
* `enable_hls`
* `stop_on_no_reader`

### 5.6 `publish`

仅 `stream_bridge` / `file_transcode` 使用，`kind` 只允许：

* `file`
* `udp_mpegts_multicast`
* `rtp_multicast`

### 5.7 `record`

* `enabled`
* `format = mp4 | hls | both`
* `segment_sec`
* `save_path`（已忽略，录制目录由系统托管）
* `as_player`
* `archive_policy`
* `retention_days`

ZLM 原生就有 `startRecord/stopRecord` 和 `on_record_mp4`，也有 `getMp4RecordFile` 可列出已有录像。([ZLMediaKit][7])

### 5.8 `recovery`

* `policy = never | auto`
* `resume_mode = from_beginning | from_offset | reattach_only`
* `backoff`
* `max_consecutive_failures`

### 5.7 `schedule`

* 立即启动
* 定时启动
* 循环计划
* 开机恢复
* 时间窗口
* 冷却时间

### 5.8 `resource`

* `required_labels`
* `preferred_labels`

## 6. 推荐默认值

参考 ZLM 配置层默认项，系统建议底层默认采用：`modify_stamp=2`、`enable_audio=1`、`add_mute_audio=1`、`auto_close=0`、`continue_push_ms=15000`、`mp4_as_player=0`、`mp4_max_second=3600`；ZLM 的 `[multicast]` 还给了 `addrMin/addrMax/udpTTL`，HTTP API 也可通过 `allow_ip_range` 做白名单。([ZLMediaKit][6])

在此基础上，北向系统建议收敛成更适合业务系统的默认值：

* `record.enabled = false`
* `expose.enable_rtsp = true`
* `expose.enable_rtmp = true`
* `expose.enable_http_ts = true`
* `expose.enable_http_fmp4 = true`
* `expose.enable_hls = false`
* `recovery.policy = auto`

### 6.2 `archive_quality`

归档档：优先 `copy`，失败再转 H.265/H.264 + AAC。

### 6.3 `multicast_ts`

固定组播节目档：MPEG-TS over UDP，默认 `pkt_size=1316`、`reuse=true`、`ttl=1`。

### 6.4 `rtmp_hevc_ext`

仅在上下游明确支持 RTMP-H265 扩展时启用，不作为默认档。([ZLMediaKit][8])

## 7. 任务执行链路

### 7.1 `live_relay`

标准链路：

`VALIDATE → ReserveStream → addStreamProxy → WaitMediaOnline → startRecord(Optional) → addStreamPusherProxy(Optional) → Monitor → stopRecord → close_streams`

`addStreamProxy` 支持 RTSP/RTMP/HLS/HTTP-TS/HTTP-FLV 拉流代理，但 payload 稳定集是 H264/H265/AAC/G711/Opus；`rtp_type` 对 RTSP 可选 TCP/UDP/Multicast；停止建议走 `close_streams`，因为 `close_stream` 已经过期；若还需要主动向外推流，可叠加 `addStreamPusherProxy`。([ZLMediaKit][7])

### 7.2 `file_transcode`

标准链路：

`ProbeInput → BuildFFmpegPlan → LaunchFFmpeg → ProgressWatch → ArtifactCommit → Success/Fail`

这类任务**不走 ZLM 主链路**。
虽然 ZLM 有 `addFFmpegSource`，但它暴露的控制面只有 `src_url / dst_url / timeout_ms / enable_hls / enable_mp4 / ffmpeg_cmd_key` 这一层，远达不到“任务级细参数 + 默认值 + 恢复 + 审计”的要求，所以只把它留作调试能力，不作为系统主执行器。([ZLMediaKit][7])

### 7.3 `file_to_live`

默认链路：

`ProbeInput → FFmpeg push to local ZLM → WaitMediaOnline → startRecord(Optional) → Monitor`

高级链路：

`ProbeInput → FFmpeg tee 输出到 (ZLM ingest + 文件)`

FFmpeg `tee` muxer 可以把同一份编码后的音视频同时写到多个输出，且只编码一次；`use_fifo=1` 可以把各个 slave output 放到独立线程；`onfail=ignore` 可以让某个输出失败时不影响其他输出。([FFmpeg][10])

### 7.4 `multicast_bridge`

这是 V1 的一级能力，不是备注字段。

#### A. RTSP multicast 输入

直接用 `addStreamProxy(rtp_type=2)`。
ZLM 文档里 `rtp_type=2` 就是 multicast；总览文档也明确写 RTSP 支持 `rtp multicast`。([ZLMediaKit][7])

#### B. 原始 UDP MPEG-TS / RTP multicast 输入

默认走 `FFmpeg → 本地 ZLM ingest`。
原因是 README 把“RTP multicast/UDP multicast ingestion and conversion”列在闭源 IPTV 版里，所以开源版设计不把它当作既有能力；而 FFmpeg 的 UDP 协议层已支持 `localaddr`、`reuse`、`ttl`、`pkt_size`、`dscp` 等参数。([GitHub][11])

#### C. RTSP 会话组播输出

走 ZLM 原生 RTSP multicast。
配置文件 `[multicast]` 提供 `addrMin`、`addrMax`、`udpTTL`，适合“客户端通过 RTSP 协商进组播”的模式。([ZLMediaKit][6])

#### D. 固定组地址持续输出

走 FFmpeg `udp://239.x.x.x:port` 或 `rtp://239.x.x.x:port`。
这里我故意**不**把 `startSendRtp` 当成主方案，因为 ZLM 文档把它定义为“GB28181 客户端 ps-rtp 推流”，更像国标/RTP 发送接口，而不是固定节目多播分发接口。这个是设计上的保守约束。([ZLMediaKit][7])

### 7.5 `rtp_receive`

标准链路：

`openRtpServer → WaitData → Monitor → on_rtp_server_timeout / closeRtpServer`

`openRtpServer` 会创建 GB28181 RTP 接收端口；超时后可自动释放；`listRtpServer` 能用于恢复期对账。([ZLMediaKit][7])

## 8. 并发、调度与状态机

统一状态机：

`CREATED → VALIDATING → QUEUED → DISPATCHING → STARTING → RUNNING → STOPPING → SUCCEEDED | FAILED | CANCELED | LOST | RECOVERING`

调度模型：

* `media-core` 用 PostgreSQL 做真相库；
* `task_leases` 控制租约，防止双执行；
* `media-agent` 定期发心跳；
* 每次执行形成一条 `task_attempt`；
* 资源按 slot 分配，而不是裸线程数。

建议 slot 维度：

* `zlm_proxy_slots`
* `ffmpeg_cpu_slots`
* `ffmpeg_gpu_slots`
* `record_io_slots`
* `multicast_slots_per_interface`

Tokio 只负责高并发控制流、Hook 接收、API 服务，不负责媒体编码；因为 Tokio 官方明确说它主要面向 I/O bound 应用，纯 CPU 并行工作建议用其他方式处理。这里我们直接把 CPU 密集媒体处理下放到外部 FFmpeg 进程。([Tokio][4])

## 9. 自动恢复设计

恢复遵循 **“DB 真相 + ZLM 现场 + Agent 现场”** 三方对账。

### 9.1 恢复触发源

* `media-core` 重启
* `media-agent` 重启
* ZLM 重启
* 宿主机重启
* FFmpeg 进程异常退出

### 9.2 恢复观测点

* ZLM `on_server_started`：可直接作为“ZLM 节点刚重启”的事件；
* ZLM `on_server_keepalive`：默认 10 秒一次，可做活性心跳；
* `getThreadsLoad` / `getWorkThreadsLoad`：看线程负载与延迟；
* `getStatistic`：看主要对象数量；
* `getMediaList`：看内部流是否仍在线；
* `listRtpServer`：看 RTP 接收端口是否仍存在。([ZLMediaKit][12])

### 9.3 恢复规则

#### `live_relay`

若 `getMediaList` 还能查到内部流，则直接 reattach；查不到则重新发起 `addStreamProxy`。([ZLMediaKit][7])

#### `file_transcode`

若 agent 仍能找到存活的 FFmpeg 进程，则接管日志与状态；否则根据 `resume_mode` 从头重跑或判失败。

#### `file_to_live`

同时检查 FFmpeg 进程和 ZLM 内部流；缺任一侧就重启整条链。

#### `rtp_receive`

对账 `listRtpServer`；若端口丢失则重开。([ZLMediaKit][7])

### 9.4 无观众与按需拉流

`on_stream_none_reader` 可用于“无人观看自动关断”；`on_stream_not_found` 可用于按需拉流。
但要注意两个关键语义：

1. `on_stream_none_reader` 里，**MP4/HLS 录制默认不算 viewer**；
2. 若 `auto_close=1`，ZLM 会直接关流，不再触发无观众 Hook。

所以系统默认应保持 `auto_close=0`，并且对“只录制不播放”的任务，要么把 `mp4_as_player=1`，要么在控制面把“录制中”本身视为保活条件。([ZLMediaKit][12])

## 10. 数据库模型

生产建议 PostgreSQL；开发/单机演示可用 SQLite。SQLx 同时支持这两类后端。([Docs.rs][13])

主表建议如下：

### `media_nodes`

节点注册表。
关键字段：`id, hostname, labels, zlm_api_base, agent_rpc_addr, network_mode, interfaces, healthy, last_seen_at`

### `node_capabilities`

能力矩阵。
关键字段：`node_id, ffmpeg_protocols, ffmpeg_formats, ffmpeg_encoders, ffmpeg_decoders, zlm_api_list, zlm_version`

### `tasks`

任务主表。
关键字段：`id, type, status, requested_spec(jsonb), resolved_spec(jsonb), idempotency_key, priority, assigned_node_id, created_at, updated_at`

### `task_attempts`

任务执行尝试。
关键字段：`id, task_id, attempt_no, node_id, worker_kind, status, pid, zlm_key, zlm_stream, rtp_port, start_at, end_at, exit_code, checkpoint_json`

### `task_leases`

调度租约。
关键字段：`task_id, holder, lease_token, expires_at`

### `stream_bindings`

内部流与底层对象绑定。
关键字段：`task_id, schema, vhost, app, stream, zlm_proxy_key, zlm_pusher_key, rtp_stream_id`

### `record_files`

录像目录。
关键字段：`task_id, attempt_id, vhost, app, stream, file_path, file_size, time_len, start_time, source=hook`

### `task_events`

事件流水。
关键字段：`task_id, attempt_id, source(zlm_hook|zlm_api|agent|ffmpeg|core), event_type, payload, created_at`

### `task_checkpoints`

可恢复任务的检查点。
关键字段：`task_id, offset_ms, segment_no, extra_json`

索引重点放在：

* `tasks(status, priority, created_at)`
* `tasks(idempotency_key)`
* `task_attempts(task_id, attempt_no desc)`
* `stream_bindings(schema, vhost, app, stream)`
* `record_files(task_id, start_time desc)`

## 12. 底层适配映射

### 12.1 ZLM Adapter

适配层负责把任务翻译成 ZLM 调用：

* `live_relay` → `addStreamProxy`
* `relay + record` → `addStreamProxy + startRecord`
* `主动外推` → `addStreamPusherProxy`
* `停止` → `stopRecord + close_streams`
* `RTP 接收` → `openRtpServer / listRtpServer / closeRtpServer`
* `PS-RTP 外发` → `startSendRtp`
* `录像入库` → `on_record_mp4`
* `会话调试` → `getAllSession / kick_session / kick_sessions / getMediaPlayerList`
* `截图` → `getSnap`

这些接口都在官方 REST/WebHook 文档里。([ZLMediaKit][3])

### 12.2 FFmpeg Supervisor

FFmpeg 统一由 agent 以“计划执行”方式启动，而不是让上层传裸命令。
内部执行对象建议叫 `ExecutionPlan`，包含：

* `inputs[]`
* `filters[]`
* `outputs[]`
* `runtime_policy`
* `progress_parser`
* `artifact_collectors`

这样就能实现：

* 结构化参数转 FFmpeg 命令；
* 同一逻辑支持 CPU/GPU 两套后端；
* 重启时可恢复命令上下文；
* UI 上可显示“计划视图”和“最终命令视图”。

### 13.2 流中心

### 13.3 组播中心

## 14. 安全设计

安全上有 5 条硬规则：

1. **ZLM API 不直接暴露给上层业务**，统一走 `media-core`。
2. **ZLM `secret` 必须启用**，并结合 `allow_ip_range` 把 HTTP API 限到内网或指定网段。([ZLMediaKit][14])
3. **不要在生产暴露 `hook.admin_params`**。文档明确说明它匹配后可绕过播放/推流鉴权，适合开发调试，不适合生产。([ZLMediaKit][15])
4. **上层不允许传原始 FFmpeg 命令**。只允许传结构化字段，避免命令注入和不可恢复任务。
5. **文件路径必须做 allowlist**。例如只允许 `/data/media/**`、`/mnt/media/**`、对象存储挂载目录。

内部链路建议：

* `media-core ↔ media-agent`：mTLS
* 用户到 `media-core`：JWT / OAuth2 / SSO
* Hook 回调：固定源 IP + shared secret + 幂等 event_id
* 审计：所有任务创建、修改、终止都写操作日志

## 15. 部署与打包

### 15.1 标准部署

首版推荐 **Docker Compose**，4 个服务：

* `media-core`
* `media-agent`
* `zlmediakit`
* `postgres`

ZLMediaKit 官方仓库已经给了 Docker image 和直接 `docker run` 示例，也可基于 Dockerfile 自行构建。([GitHub][11])

目录建议：

* `/data/zlm/www`
* `/data/zlm/record`
* `/data/media/work`
* `/data/media/logs`
* `/data/postgres`

### 15.2 组播部署

涉及 UDP 组播时，推荐两种网络模式：

#### 方案 A：`host` 网络

优先推荐。
Docker 官方文档说明 `host` 模式下容器共享宿主机网络命名空间，没有独立 IP，也不走 `-p` 端口映射；它适合性能敏感和需要处理大量端口的场景。对组播来说，这通常是最省事的。([Docker Documentation][16])

#### 方案 B：`macvlan`

当网络团队要求“容器直接出现在物理网络、拥有独立二层身份”时用 `macvlan`。Docker 官方文档说明它会让容器看起来像直接连到物理网络，需要指定 parent 网卡；但它只支持 Linux，很多云厂商会阻断，而且容器默认不能直接和宿主机通信。([Docker Documentation][17])

#### 不推荐：`ipvlan l3`

Docker 官方文档明确写了 **IPvlan L3 会丢弃所有 broadcast 和 multicast**，因此不适合你的组播节点。([Docker Documentation][18])

所以部署建议很明确：

* **普通 TCP/HTTP 任务**：Compose 默认 bridge 即可；
* **有 UDP 组播**：`media-agent + ZLM` 用 `host`，或在强网络隔离要求下用 `macvlan`；
* **云主机**：优先 `host`，别默认上 `macvlan`。

## 16. 支持矩阵

V1 的支持策略不是“宣称所有协议都能任意互转”，而是分两层：

### 16.1 ZLM 原生稳定主航道

* 实时接入/分发：RTSP / RTMP / HLS / HTTP-FLV / HTTP-TS / fMP4 / MP4 / GB28181 / RTP
* RTSP `rtp multicast`
* RTP/GB 接收与 PS-RTP 发送
* 录制、Hook、API、会话调试
  这部分稳定编码集以 **H264/H265/AAC/G711/OPUS** 为主，部分路径含 MJPEG；其他编码可转发但不能保证协议转换。([ZLMediaKit][1])

### 16.2 FFmpeg 扩展能力面

* 离线转码 / 转封装 / 拼接 / 字幕 / 水印 / 抽帧
* 文件转直播
* 固定 UDP 组播输入/输出
* 更宽的协议、封装、编解码支持
 真正的支持面由节点上的 FFmpeg build 决定，因此 agent 启动时必须做 capability probe。([FFmpeg][2])

额外可扩展但不作为 V1 核心承诺的能力：

* SRT
* GPU 硬编和对象存储归档
  这部分能力不纳入当前控制面实现范围。

## 17. V1 开发顺序

建议按下面顺序推进，风险最低：

### 阶段 1：单节点最小闭环

* 录像目录
* 会话调试

### 阶段 4：增强项

* GPU Worker
* MinIO/S3 归档
* 多节点调度
* SRT 深化

## 18. 最终定案摘要 ✅

这套方案的核心不是“Rust 去搬运码流”，而是：

* **Rust 做统一控制面**
* **ZLMediaKit OSS 做实时媒体面**
* **FFmpeg 做复杂处理与组播补齐**
* **PostgreSQL 做恢复锚点**
* **Compose / host-network / macvlan 做易部署**

这样既满足你现在的 6 条要求，也给后续扩展留足空间，而且**不会踩到“开源版其实没有这个能力”的坑**。
下一步最自然的工作，就是把这份设计直接展开成 **数据库 DDL + OpenAPI 接口草案 + docker-compose 初稿**。
