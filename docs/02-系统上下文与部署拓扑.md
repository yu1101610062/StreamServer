# 02. 系统上下文与部署拓扑

## 1. 文档目标

本文件定义系统组件边界、默认部署方式、网络与信任边界，以及开发和联调时必须遵守的拓扑假设。

## 2. 系统上下文

```text
业务系统 / Web 管理台
          |
          | HTTPS + JWT
          v
     media-core
      |   |   |
      |   |   +--> PostgreSQL
      |   |
      |   +------> ZLM Hook Receiver
      |
      +----------> Agent Control Stream (gRPC + mTLS)
                       |
                       +--> media-agent
                               |--> FFmpeg / ffprobe
                               +--> local ZLMediaKit
```

## 3. 组件职责

| 组件 | 主要职责 | 不负责 |
| --- | --- | --- |
| `media-core` | API、鉴权、模板、参数解析、调度、恢复、审计、前端后端 | 不直接拉流、不直接编解码 |
| `media-agent` | 节点执行、能力探测、日志采集、进程接管、NIC 绑定 | 不做租户权限和全局调度 |
| ZLMediaKit | 实时拉流代理、分发、录制、RTP/GB 接收、Hook 发射 | 不做业务状态机 |
| FFmpeg/ffprobe | 探测、转码、文件转直播、持续组播输出 | 不承载上层任务语义 |
| PostgreSQL | 真相库、状态机持久化、租约、事件与审计 | 不做缓存层或消息队列 |

## 4. 默认部署拓扑

### 4.1 本地开发拓扑

- 单机 native 部署。
- `media-core`、`postgres`、`zlmediakit`、`media-agent` 同机，由 systemd 管理。
- 普通 HTTP/RTSP/RTMP 联调直接使用宿主机端口。
- 涉及 UDP 组播时，`media-agent` 和 `zlmediakit` 直接绑定宿主机网卡。

### 4.2 单站点生产拓扑

- 一个 `media-core` 实例即可；首版默认不做控制面多活。
- 一个 PostgreSQL 主库。
- 多个媒体节点，每个节点至少包含 `media-agent + ZLMediaKit + FFmpeg`。
- 所有节点位于同一受信局域网或受控专网内。

节点发现与调度规则：

- 节点不互相发现；每个 `media-agent` 只主动连接中心 `media-core`。
- 节点唯一身份只由 `node_id` 决定，`node_name` 不参与身份判定。
- 对于带源流地址的网络输入，Core 调度时优先用该源地址匹配节点已上报网卡；任一网卡命中同网段即可优先。
- 同网段只是调度加分项，不是强约束；若没有命中节点，仍回落到其他在线节点。
- 若多个节点同样命中或都未命中同网段条件，再按节点当前 `slot_usage` 和 `running_tasks` 选择更轻载的节点。

### 4.3 组播拓扑

- 优先 `host` 网络，直接使用宿主机网卡加入和发送多播。
- 若网络隔离要求强，优先通过独立物理网卡、VLAN 或主机路由策略隔离。
- 禁止把组播节点放到会丢弃广播或组播的虚拟网络路径中。

## 5. 网络端口基线

| 服务 | 默认端口 | 访问方 | 说明 |
| --- | --- | --- | --- |
| `media-core` HTTP | `8080` | 前端、业务系统、内网调试 | 开发环境默认 HTTP |
| `media-core` HTTPS | `8443` | 前端、业务系统 | 生产默认入口 |
| `media-core` internal gRPC | `50051` | `media-agent` | 双向控制流入口 |
| PostgreSQL | `5432` | `media-core` | 内网访问 |
| ZLM HTTP API | `8081` | `media-core`、调试工具 | 仅内网开放 |
| ZLM 媒体端口 | 由配置决定 | 播放/推流客户端 | 按协议开放 |

说明：

- 端口是默认值，不是硬编码；实现中必须支持配置覆盖。
- 只有 `media-core` 能访问 ZLM API；业务方不得绕过控制面访问 ZLM。

## 6. 信任边界

### 6.1 外部边界

- 业务调用方与 Web 管理台只能访问 `media-core`。
- ZLM 原始 API、Agent 控制流、PostgreSQL 不对业务侧开放。

### 6.2 内部边界

- `media-core <-> media-agent` 使用 mTLS。
- `media-core <-> ZLM` 使用 shared secret + 源地址白名单。
- ZLM Hook 只接受白名单源地址和签名校验。

## 7. 存储与挂载基线

| 路径 | 用途 | 所属组件 |
| --- | --- | --- |
| `/data/zlm/www` | ZLM 静态资源与 HLS 输出 | ZLM |
| `/data/media/work` | FFmpeg 中间文件和工作目录 | Agent |
| `/data/media/logs` | Agent 本地滚动日志 | Agent |
| `/data/postgres` | PostgreSQL 数据目录 | PostgreSQL |

## 8. 部署约束

- 仅 Linux 节点允许承载 `media-agent` 与 ZLM。
- 首版不支持 Windows 节点作为正式执行节点。
- 所有执行节点必须具备稳定的本地磁盘写入能力。
- 需要组播的节点必须提前确认交换机、网卡、主机防火墙和路由策略。

## 9. 配置分层

配置分 4 层，自低到高覆盖：

1. 编译时默认值。
2. 环境配置文件。
3. 环境变量。
4. 数据库中的模板/系统配置。

运行中的任务一旦进入 `resolved_spec`，不再回读动态默认值。
