# 03. 领域模型与状态机

## 1. 文档目标

本文件定义 StreamServer 的核心实体、字段语义、状态迁移、操作限制和规格解析规则。实现中不允许出现与本文件冲突的隐式状态。

## 2. 主键与命名规则

- 所有主键统一由 `media-core` 生成 `UUIDv7`。
- 时间字段统一使用 `timestamptz`，序列化为 RFC 3339。
- 对外字段统一使用 `snake_case` JSON 命名。
- 任务显示名 `name` 可重复，任务 ID 唯一。

## 3. 核心实体

| 实体 | 作用 | 关键字段 |
| --- | --- | --- |
| Task | 任务主对象 | `id`, `type`, `status`, `requested_spec`, `resolved_spec`, `priority`, `assigned_node_id` |
| TaskAttempt | 任务的一次执行尝试 | `attempt_no`, `worker_kind`, `status`, `node_id`, `pid`, `zlm_key`, `exit_code` |
| TaskLease | 排他执行租约 | `task_id`, `holder`, `lease_token`, `expires_at` |
| StreamBinding | 任务与底层流对象绑定关系 | `schema`, `vhost`, `app`, `stream`, `zlm_proxy_key`, `zlm_pusher_key` |
| RecordFile | 录像或离线产物目录项 | `file_path`, `file_size`, `time_len`, `source` |
| TaskEvent | 统一事件流水 | `source`, `event_type`, `payload`, `attempt_no` |
| TaskCheckpoint | 可恢复任务检查点 | `offset_ms`, `segment_no`, `extra_json` |
| MediaNode | 执行节点 | `node_name`, `labels`, `healthy`, `last_seen_at` |
| NodeCapability | 节点能力矩阵 | `ffmpeg_protocols`, `ffmpeg_encoders`, `zlm_api_list` |

## 4. 任务规格模型

### 4.1 统一结构

```json
{
  "type": "stream_ingest",
  "name": "relay-camera-01",
  "priority": 50,
  "common": {},
  "input": {},
  "process": {},
  "publish": {},
  "record": {},
  "recovery": {},
  "schedule": {},
  "resource": {}
}
```

### 4.2 解析顺序

`resolved_spec` 的生成顺序固定为：

1. 载入任务类型系统默认值。
2. 叠加用户提交的 `requested_spec`。
3. 执行字段标准化和派生值填充。
4. 写入数据库并冻结。

规则：

- 后一步永远覆盖前一步。
- 任务进入 `QUEUED` 后，`resolved_spec` 不可再被修改。
- `requested_spec` 永远按原样留档，不做回写。

## 5. 任务状态机

### 5.1 任务状态定义

| 状态 | 含义 |
| --- | --- |
| `CREATED` | 已建单，尚未完成解析或校验 |
| `VALIDATING` | 校验参数、模板和能力要求 |
| `QUEUED` | 已就绪，等待调度 |
| `DISPATCHING` | 正在获取租约并下发到节点 |
| `STARTING` | 节点已接单，正在启动底层执行对象 |
| `RUNNING` | 执行中 |
| `STOPPING` | 收到停止或取消，正在回收底层对象 |
| `RECOVERING` | 系统或节点重启后正在对账和接管 |
| `SUCCEEDED` | 成功完成 |
| `FAILED` | 执行失败且不再重试 |
| `CANCELED` | 被人工取消 |
| `LOST` | 节点或底层对象失联，等待人工或自动恢复 |

### 5.2 允许迁移

```text
CREATED -> VALIDATING -> QUEUED -> DISPATCHING -> STARTING -> RUNNING
RUNNING -> STOPPING -> SUCCEEDED
RUNNING -> STOPPING -> CANCELED
RUNNING -> FAILED
RUNNING -> LOST
LOST -> RECOVERING -> RUNNING
LOST -> RECOVERING -> FAILED
FAILED -> QUEUED       (仅 retry 生成新 Attempt)
SUCCEEDED -> QUEUED    (仅 clone 生成新 Task)
CANCELED -> QUEUED     (仅 clone 生成新 Task)
```

约束：

- `FAILED`、`SUCCEEDED`、`CANCELED` 为终态。
- `retry` 不修改原 Attempt，而是在同一 Task 下创建新 Attempt。
- `clone` 必须生成新 Task ID。

## 6. Attempt 状态

| 状态 | 含义 |
| --- | --- |
| `PENDING` | 尝试已创建，尚未分派 |
| `STARTING` | 节点执行器正在启动 |
| `RUNNING` | 进程或 ZLM 对象已在线 |
| `STOPPING` | 正在停止 |
| `SUCCEEDED` | 本次尝试成功 |
| `FAILED` | 本次尝试失败 |
| `ADOPTED` | 经恢复流程接管到已有底层执行对象 |
| `ORPHANED` | 底层对象存在但控制链断开，待接管 |

## 7. 操作限制矩阵

| 操作 | 允许状态 |
| --- | --- |
| `start` | `CREATED`, `FAILED`, `CANCELED` |
| `stop` | `DISPATCHING`, `STARTING`, `RUNNING`, `RECOVERING`, `LOST` |
| `cancel` | `CREATED`, `VALIDATING`, `QUEUED`, `DISPATCHING`, `STARTING`, `RUNNING`, `RECOVERING` |
| `retry` | `FAILED`, `LOST` |
| `clone` | `SUCCEEDED`, `FAILED`, `CANCELED`, `LOST` |

说明：

- `start` 用于显式启动非自动启动任务；自动启动任务由创建流程直接进入 `VALIDATING`。
- `cancel` 与 `stop` 的区别是：`cancel` 表示不再恢复且不保留继续执行意图。

## 8. 统一事件模型

每个事件必须包含：

- `task_id`
- `attempt_no`
- `source`
- `event_type`
- `event_level`
- `event_time`
- `payload`

`source` 固定值：

- `core`
- `agent`
- `ffmpeg`
- `zlm_api`
- `zlm_hook`
- `scheduler`
- `user`

## 9. 任务类型特殊规则

### 9.1 `live_relay`

- 判定成功的标志是内部流在线。
- 若启用录制，录制启动失败不回滚主流，任务进入 `RUNNING` 并打出降级事件。

### 9.2 `file_transcode`

- 判定成功的标志是 FFmpeg 退出码为 0 且产物校验通过。
- 不依赖 ZLM 存活。

### 9.3 `file_to_live`

- 判定成功的标志是 FFmpeg 推流在线且 ZLM 内部流可见。
- 任一侧丢失都进入 `LOST` 或 `FAILED`。

### 9.4 `multicast_bridge`

- 若模式为输入桥接，成功标志是桥接后的内部流在线。
- 若模式为持续输出，成功标志是 FFmpeg 已成功发送且输出端无启动错误。

### 9.5 `rtp_receive`

- 判定成功的标志是 `openRtpServer` 成功并收到有效媒体数据。
- 超时由 ZLM Hook 进入 `LOST` 或 `FAILED`。

## 10. 租约规则

- 任一 Task 在同一时刻只允许一个有效租约。
- 租约默认 30 秒有效，每 10 秒由持有方续租。
- 租约续期失败 3 次后，任务进入 `LOST`。
- 恢复流程必须先夺回租约，再执行接管或重建。
