## 文档分工

| 文档 | 目标读者 | 作用 |
| --- | --- | --- |
| `设计文档.md` | 架构负责人、评审人 | 说明总体方案、关键选型和能力边界 |
| `01-03` | 产品、后端、前端、测试 | 锁定范围、术语、领域模型和状态语义 |
| `04-07` | 后端、Agent、适配层 | 锁定接口契约、执行模型和第三方集成边界 |

## 开工前完成定义

以下条件全部满足后，才视为进入正式开发阶段：

- 总体设计和拆分文档已经过一次团队评审。
- `TaskSpec`、状态机、数据库表结构、北向 API、Agent RPC 不再存在未决字段。
- 组播网络方案、测试样本、镜像来源、目录挂载、证书策略已确认。
- 任务类型的验收标准已经写入 [10-测试计划与验收标准](docs/10-测试计划与验收标准.md)。
- 代码目录、分支策略、迁移方式、日志规范已在 [12-研发流程与协作规范](docs/12-研发流程与协作规范.md) 固化。

## 维护规则

- 新增能力前，先更新对应文档，再改代码。
- 接口、状态、表结构变更必须同步更新 `03`、`04/05`、`08` 三份文档。
- 若实现与文档冲突，以文档评审后的最新版本为准；未更新文档的实现不视为完成。
